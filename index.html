What is Functional Programming?

Programming style that treats computation as execution of mathematical functions.


Typically composed of two core concepts:

  • Higher order functions
  • Avoiding state

No quick and exact definition of Functional Programming. Typically defined as a style of programming alternative to imperative style. Two core concepts are generally followed - higher order functions and avoiding state (immutability).

Higher Order Functions

Functions that return a function as an output and/or take one or more functions as an input.

Example (functions as input): 


def example(): Unit = {
  def sum(a: => Int, b: => Int): Int = {
    a + b
  }

  sum(1, 2)
  sum("two".length, "three".length)
}

First-Class Functions

Functions are first-class citizens - operations typically supported for objects are also supported for functions


Functions can be passed as arguments, returned as output, assigned to variables, and stored in data structures


Common language support includes: JavaScript, C#, Java, Scala, Python, etc.

Avoiding State - Immutable Objects

Immutable objects are objects whose state cannot be modified after they are created.


Beneficial since developers do not need to reason about or depend on complex state changes. Functions outputs become predictable.


Immutable objects are inherentily thread-safe!

Standard OOP practices involve creating mutable objects - many setter properties, methods that directly modify state of objects, etc. Reasoning about output from a function is considerably easier and predictable - outside world cannot modify object state, and affect function output. Immutable objects are automatically thread-safe. Need for locking is completely eliminated. Drawback - real world becomes considerably more difficult to implement using immutable objects. Less efficient, since every time a state would be modified, a new object is created.

Functional Programmng Languages

Languages that support/encourage FP concepts:

  • Haskell (most common purely functional language)
  • .NET (C# and F#)
  • Java (esp. Java 8)
  • JavaScript
  • Scala, Erlang, Lisp

Note: Functional Programming != Functional Programming Language

Haskell vs. Scala vs. JavaScript

List of some common FP languages. Haskell is the most common purely functional language. The rest of the languages on the list are "hybrid" - they support a combined imperative/functional approach. FP != FP Language Most FP languages are hybrid, which means they allow for flexibility in writing imperative code where it makes sense, but they support writing pure functional code as well.

Why should you use FP?

  • To learn another programming style
  • Writing concurrent or parallel code (threading)
  • Writing highly scalable distributed code
  • Reduce defect count through immutability and brevity
  • Increased modularity, easier testing
  • Write less code

Pitfalls?

Immutability Challenges

Representing real world while using immutability is challenging.


UI code is not well suited for functional programming


Immutability results in object copying on every change. In worst case, copying of entire application state may required.

Pitfalls?

Performance

Immutability - copy objects instead of changing state


Recursion less efficient due to higher memory requirements


Recursion can lead to stack overflow unless it is tail-call optimized.

Recursion stack overflow example - purely functional implementation of foldRight. The recursive call happens at the beginning of the method (vs. at the end!). Scala implements foldRight using a reverse and foldLeft!

Pitfalls?

Language Maturity

Languages typically used for functional programming are not as mature as .NET/Java/C


IDE support is far from perfect (i.e. Scala).


FP not nearly as widely used as imperative/OOP

Pitfalls?

Fewer resources to get help from

Majority of functional programming users are academic


Far fewer resources to rely on compared to standard OOP languages.


Stack Overflow: C# (616k), Java (606k), JavaScript (578k), Scala (24k), Haskell (15k), Erlang (4k), functional-programming(5k)

Fewer real-world (especially Enterprise) examples of using functional programming. Enterprise clients typically do not want to pay for unproven technology, even if it means missing out on "latest and greatest" improvements.

Pitfalls?

Learning Curve & Readability

OOP traditionally taught and used in workforce. FP requires a different programming mentality most developers not used to.


Brevity is a double edged sword - easy to fall into a trap of writing concise code that cannot be undersood easily


Relying on symbols instead of descriptive function names commonplace in FP, making it tougher to understand for new developers.

Scala

Object-Oriented Meets Functional

Hybrid OOP + Functional Programming JVM language.

"Seamless" Java Interop - leverage an immense repository of Java libraries


Used by Twitter, LinkedIn, Intel, etc.

Scala Highlights

  • Functional and Object Oriented
  • Highly Scalable
    • Native support for threading and concurrency
    • Can be used as a scripting language
    • REPL
    • Staticly Typed
  • Strong tuple support (limited 22 params)

Scala Highlights

Case Classes

  • Immutable by default (mutability supported)
  • Eliminates need for DTOs
  • Pattern matching
  • Auto-generated methods
    • copy
    • Sensible toString
    • equals by-value (instead of by ref)
  • Abstract Data Types

Scala Highlights

Type System

Extremely powerful type system; can be extremely complex.


  • Support for standard classes and abstract classes
  • Does not require public classes to be defined in filenames of same name
  • No support for static methods/classes
    • Uses object modifier to declare class as a singleton
  • Robust support for generics, incl. type variance


This only scratches the surface of the Scala type system...

Scala Highlights

Traits (mixins)

  • Like interfaces ... but not quite
  • Allow abstract and full method definitions
  • Cannot contain constructors
  • Eliminate diamond pattern - compiler knows which method to call in case of naming conflicts
  • self types - access properties of class a trait is mixed into
  • Can mix in multiple traits to a trait/class
  • Traits can be mixed in at variable declaration time:
    • var p = new Person with Job

Scala Highlights

Pattern Matching - Case Class


trait Geometry
case class Point(x: Int, y: Int) extends Geometry
case class Triangle(p1: Int, p2: Int, p3: Int) extends Geometry
case class Square(x0: Int, x1: Int, y0: Int, y1: Int) extends Geometry

def matchIt(g: Geometry): Unit = g match {
  case Point(0,0) => println(s"Origin")
  case Point(x, y) => println(s"Point ($x, $y)")
  case t:Triangle => println(s"Triangle: $t")
  case Square(_, _, _, _) => println("Square")
  case Circle(_, _, _) => println("Circle")
}

matchIt(Point(0,0))      // Outputs: Origin
matchIt(Point(1,2))      // Outputs: Point (1, 2)
matchIt(Triangle(1,2,3)) // Outputs: Triangle(1,2,3)

Scala Highlights

Pattern Matching - Lists


def matchIt(l: List[_]): Unit = l match {
  case Nil => println("Empty List")
  case x :: Nil => println("Single Element")

matchIt(Nil)         // Outputs: Empty List

Concurrency using Futures

Future comprehension - parallel requests


val googleResponse: Future[Response] = WS.url("http://google.com").get().map(getLatency)
val msoftResponse: Future[Response] = WS.url("http://microsoft.com").get().map(getLatency)
val soResponse: Future[Response] = WS.url("http://stackoverflow.com").get().map(getLatency)

// If any of the futures fail, minResponse will also be a failed future
val minResponse: Future[Long] = for {
  googleTime <- googleResponse
  msoftTime <- msoftResponse
  soTime <- soResponse
} yield {
  List(googleTime, msoftTime, soTime).min
}

// Nothing printed in case of a failure!
minResponse.onSuccess { case time => println(s"Minimum Response Time: $time")}

Quicksort - Traditional Implementation


def sort(xs: Array[Int]) {
  def swap(i: Int, j: Int) {
    val t = xs(i); xs(i) = xs(j); xs(j) = t
  }

  def sort1(l: Int, r: Int) {
    val pivot = xs((l + r) / 2)
    var i = l; var j = r
    while (i <= j) {
      while (xs(i) < pivot) i += 1
      while (xs(j) > pivot) j -= 1
      if (i <= j) {
        swap(i, j)
        i += 1
        j -= 1
      }
    }
    if (l < j) sort1(l, j)
    if (j < r) sort1(i, r)
  }

  sort1(0, xs.length - 1)
}

Quicksort - Functional Programming Implementation


// No state modified - array is copied, never modified!
def sort(xs: Array[Int]): Array[Int] = {
  if (xs.length <= 1) {
    xs
  } else {
    val pivot = xs(xs.length / 2)
    Array.concat(
      sort(xs.filter(pivot > _)),
      xs.filter(pivot == _),
      sort(xs.filter(pivot < _)))
  }
}
